## 🏗️ 架构哲学

AgentOS采用**五维正交设计原则**，从五个维度定义系统的核心价值：

1. **系统观 (System View)** — 反馈闭环、层次分解、总体设计部
2. **内核观 (Kernel View)** — 微内核、接口契约化、服务隔离、可插拔策略
3. **认知观 (Cognition View)** — 双系统协同、增量演化、记忆卷载、遗忘机制
4. **工程观 (Engineering View)** — 安全内生、可观测性、资源确定性、语义化命名、错误可追溯、文档即代码、测试隔离
5. **美学观 (Aesthetic View)** — 简约至上、极致细节、人文关怀、完美主义

完整原则文档：[ARCHITECTURAL_PRINCIPLES.md](../../agentos/manuals/ARCHITECTURAL_PRINCIPLES.md)

## 🧩 核心组件详解

### 1️⃣ 内核层 (Kernel Layer)

**职责**: 提供操作系统级的基础设施，是整个系统的基石。

**四大原子机制**:

| 机制 | 缩写 | 职责 | 核心API |
|------|------|------|---------|
| **进程间通信** | IPC | 模块间消息传递 | `agentos_ipc_send()`, `agentos_ipc_recv()` |
| **内存管理** | Mem | 分配与回收 | `agentos_mem_alloc()`, `agentos_mem_free()` |
| **任务调度** | Task | 任务生命周期管理 | `agentos_task_create()`, `agentos_task_wait()` |
| **时间服务** | Time | 定时器与时钟同步 | `agentos_timer_create()`, `agentos_time_now()` |

**设计特点**:

- ✅ **极简主义**: 仅包含4个核心机制，~25K LOC
- ✅ **高性能**: 零拷贝IPC、对象池内存分配
- ✅ **安全性**: 所有系统调用经过权限检查
- ✅ **可扩展**: 通过插件机制扩展功能

**详细文档**: [微内核架构](microkernel.md)

---

### 2️⃣ 认知层 (CoreLoopThree)

**职责**: 模拟人类认知过程，实现智能推理和决策。

**三层认知循环**:

```
┌─────────────────────────────────────────────┐
│              Action Layer (行动层)            │
│  • 工具调用                                  │
│  • 结果收集                                  │
│  • 状态更新                                  │
└────────────────────┬────────────────────────┘
                     │ 反馈
                     ▼
┌─────────────────────────────────────────────┐
│            Planning Layer (规划层)            │
│  • 任务分解                                  │
│  • 依赖排序                                  │
│  • 资源分配                                  │
└────────────────────┬────────────────────────┘
                     │ 规划结果
                     ▼
┌─────────────────────────────────────────────┐
│          Cognition Layer (认知层)             │
│  • 双系统协同 (System 1 & System 2)          │
│  • 主模型: 快速直觉推理                        │
│  • 辅模型: 深度逻辑验证                        │
└─────────────────────────────────────────────┘
```

**双系统协作机制**:

- **System 1 (主模型)**: GPT-4o-mini — 快速、低成本、高吞吐
- **System 2 (辅模型)**: GPT-4o — 深度推理、复杂决策、交叉验证

**增量演化规划**:

- 支持动态调整规划深度（默认10层）
- 失败节点自动回退，保留成功节点结果
- 基于反馈实时优化后续规划

**详细文档**: [核心循环三层](coreloopthree.md)

---

### 3️⃣ 记忆系统 (MemoryRovol)

**职责**: 实现类似人类的四层记忆卷载机制。

**四层架构**:

| 层级 | 名称 | 存储介质 | 特点 | 保持时间 |
|------|------|---------|------|---------|
| **L1** | 感觉记忆 | Redis (内存) | 原始数据、高速度 | 秒级 |
| **L2** | 短期记忆 | PostgreSQL + FAISS | 向量索引、语义检索 | 小时级 |
| **L3** | 长期记忆 | PostgreSQL | 结构化知识、关系图谱 | 天/月级 |
| **L4** | 模式记忆 | PostgreSQL | 抽象模式、经验法则 | 永久 |

**遗忘机制** (基于艾宾浩斯遗忘曲线):

```
记忆保持率 (%)
100 │╲
 80 │  ╲
 60 │   ╲_____
 40 │        ╲___
 20 │            ╲________
  0 └────────────────────────→ 时间
    0   1   2   3   7   30(天)
```

- **主动复习**: 重要记忆自动提升权重
- **被动衰减**: 未被访问的记忆逐渐降低权重
- **压缩归档**: 低权重记忆从高层向低层迁移

**详细文档**: [记忆卷载系统](memoryrovol.md)

### 4️⃣ 安全穹顶 (Cupolas)

**职责**: 为智能体系统提供企业级安全保障。

**四层防护体系**:

```
┌─────────────────────────────────────────────┐
│           Layer 4: Audit Trail (审计追踪)     │
│  全链路操作日志、不可篡改、合规报告            │
├─────────────────────────────────────────────┤
│         Layer 3: Input Sanitization (输入净化) │
│  SQL注入防护、XSS过滤、命令注入防御            │
├─────────────────────────────────────────────┤
│        Layer 2: Permission RBAC (权限控制)    │
│  基于角色的访问控制、最小权限原则               │
├─────────────────────────────────────────────┤
│         Layer 1: Workbench Sandboxing (沙箱)  │
│  进程隔离、资源限额、容器化隔离                 │
└─────────────────────────────────────────────┘
```

**核心能力**:

- ✅ **SQL注入防御率**: ≥99.9%
- ✅ **权限提升防御率**: 100%
- ✅ **攻击响应时间**: <50ms
- ✅ **攻击溯源能力**: 完整攻击链重建

**详细文档**: [安全穹顶](cupolas.md)

### 5️⃣ 守护进程层 (Daemon Layer)

**职责**: 提供长期运行的后台服务。

**守护进程列表**:

| 守护进程 | 端口 | 职责 |
|---------|------|------|
| `llm_d` | 8001 | LLM服务管理与模型调度 |
| `market_d` | 8002 | 工具市场注册与发现 |
| `monit_d` | 8003 | 系统监控与告警 |
| `sched_d` | 8004 | 任务队列与调度 |
| `tool_d` | 8005 | 工具执行引擎 |
| `gateway_d` | 8006 | API网关与负载均衡 |

**通信方式**:

- **内部通信**: 通过内核IPC机制
- **外部通信**: 通过gateway_d暴露RESTful API
- **异步消息**: Redis Pub/Sub实现事件驱动

---

### 6️⃣ 应用层 (Application Layer)

**职责**: 用户交互界面和SDK。

**主要组件**:

- **OpenLab**: Web UI，提供可视化操作界面
- **CLI Tool**: 命令行工具，适合自动化脚本
- **SDK**: 多语言绑定（Python、Go、Rust、TypeScript）

## 🔄 数据流示例

### 场景：用户提交"分析财报"任务

```
1. 用户通过OpenLab提交任务描述
   ↓
2. gateway_d接收请求，进行身份认证和输入净化(Cupolas Layer 2-3)
   ↓
3. 任务进入sched_d队列，等待调度
   ↓
4. sched_d将任务分发给CoreLoopThree
   ↓
5. Cognition Layer进行初步理解(System 1快速分析)
   ↓
6. Planning Layer制定执行计划(分解为子任务)
   ↓
7. Action Layer执行子任务:
   a. 调用tool_d获取财报数据
   b. 调用llm_d进行分析
   c. 结果写入MemoryRovol(L1→L2→L3)
   ↓
8. 各阶段结果反馈到Cognition Layer(闭环反馈)
   ↓
9. 最终结果返回给用户
   ↓
10. 审计日志记录全流程(Cupolas Layer 4)
```
