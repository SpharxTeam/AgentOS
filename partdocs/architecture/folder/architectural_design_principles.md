## 第 1 章 引言

### 1.1 编制目的

本文档为 AgentOS 提供统一的架构设计原则体系。它不是一份抽象的宣言，而是从 AgentOS 的每一行代码、每一个头文件、每一次系统调用中提炼出的设计基因。

AgentOS 是一个**面向多智能体协作的操作系统内核**——以 C11 微内核为骨骼，以三层认知循环为神经系统，以四层记忆卷载为经验基底，以安全穹顶为免疫系统。它的目标不是构建又一个 Agent 框架，而是为智能体文明建造**第一块操作系统基石**。

### 1.2 理论根基

#### 1.2.1 工程控制论——反馈即生命

> "控制系统依赖反馈来修正偏差。没有反馈的系统是盲目的。" —— 钱学森《工程控制论》

AgentOS 将反馈设计为贯穿系统每一层的核心机制：

- **实时反馈**（τ < 100ms）：行动层执行结果通过 `agentos_execution_wait` 返回认知层，修正当前任务的后续规划。这是系统条件反射级别的快速纠偏。
- **轮次内反馈**（τ = 任务周期）：增量规划器根据已完成的任务节点，动态扩展或修正后续阶段的任务 DAG。这是系统短期学习能力的体现。
- **跨轮次反馈**（τ = 会话周期）：MemoryRovol 的 L4 模式层从历史轮次中挖掘持久模式，更新认知策略参数。这是系统长期进化的根本驱动力。

控制论视角下，AgentOS 是一个**三层嵌套的负反馈系统**：每一层都有独立的感知-决策-执行-反馈闭环，同时层与层之间通过 IPC 通道和系统调用形成跨层反馈通路。

#### 1.2.2 系统工程——层次即秩序

> "复杂系统应按层次分解，每一层只关注自己层次的输入输出。" —— 钱学森《论系统工程》

AgentOS 采用严格的层次分解：

```
┌─────────────────────────────────────────────┐
│              backs/ 用户态服务                │
│     llm_d · market_d · monit_d · tool_d     │
├─────────────────────────────────────────────┤
│             dynamic/ 网关层                  │
│          HTTP · WebSocket · Stdio           │
├─────────────────────────────────────────────┤
│          syscall/ 系统调用接口                │
│    task · memory · session · telemetry      │
├─────────────────────────────────────────────┤
│            domes/ 安全穹顶                   │
│     workbench · permission · sanitizer      │
├──────────────┬──────────────────────────────┤
│ corekern/    │  coreloopthree/              │
│ 微内核基础   │  认知-执行三层循环             │
│ IPC·Mem·Task │  认知→规划→调度→执行          │
│ Time         │                              │
├──────────────┴──────────────────────────────┤
│         memoryrovol/ 四层记忆系统            │
│  L1原始卷 → L2特征层 → L3结构层 → L4模式层  │
├─────────────────────────────────────────────┤
│             utils/ 公共工具库                │
│  error · logger · metrics · trace · cost    │
└─────────────────────────────────────────────┘
```

每一层只依赖其下层提供的抽象接口，从不越级访问。这种层次纪律保证了系统各部分可以独立理解、独立测试、独立演进。

#### 1.2.3 双系统认知理论——快慢协同

> "人类思维包含两个系统：快速直觉的 System 1 和缓慢深思的 System 2。" —— 卡尼曼《思考，快与慢》

AgentOS 将双系统理论编码进架构的每一个层次：

| 层次 | System 1（快） | System 2（慢） |
|------|---------------|---------------|
| 认知层 | 辅模型快速分类与交叉验证 | 主模型深度规划与反思调整 |
| 执行层 | 执行单元快速执行 | 补偿事务验证与异常恢复 |
| 记忆层 | 吸引子网络快速模式匹配 | L4 持久同调深度模式挖掘 |
| 安全层 | 净化规则快速过滤 | 权限引擎深度策略裁决 |

双系统不是简单的速度分级，而是**认知带宽的分层管理**——System 1 处理 80% 的常规任务，System 2 集中算力于 20% 的关键决策。

#### 1.2.4 微内核哲学——极简即力量

> "内核只做内核该做的事。其他一切，都是服务。" —— Jochen Liedtke

AgentOS 遵循严格的微内核架构哲学：

- **`corekern/`** 仅包含四个不可再分的原子机制：IPC（进程间通信）、Mem（内存管理）、Task（任务调度）、Time（时间服务）。总计 7 个头文件、13 个源文件——这是整个系统的最小内核。
- **`coreloopthree/`** 和 **`memoryrovol/`** 作为内核子系统，提供认知循环和记忆演化能力。
- 所有用户态服务（LLM、市场、监控、调度、工具）运行在 `backs/` 中，通过 `syscall/` 的标准化接口与内核交互。
- 安全机制集中在 `domes/` 中，与内核逻辑解耦。

这种分离保证了：内核故障不影响服务可替换性，服务崩溃不会拖垮内核稳定性。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **原子内核 (corekern)** | 不可再分的微内核基础：IPC、内存、任务、时间 |
| **三层循环 (coreloopthree)** | 认知→规划→调度→执行的三层认知-行动运行时 |
| **记忆卷载 (memoryrovol)** | 四层递进式记忆系统：原始卷→特征层→结构层→模式层 |
| **安全穹顶 (domes)** | 虚拟工位、权限裁决、输入净化、审计追踪的安全层 |
| **系统调用 (syscall)** | 用户态服务与内核之间的标准化接口契约 |
| **守护进程 (Daemon)** | 运行在用户态的后台服务，以 `_d` 后缀命名 |
| **执行单元 (execution unit)** | 可插拔的原子执行器，通过注册表动态加载 |
| **策略接口 (strategy)** | 可运行时替换的算法抽象（规划/协同/调度/遗忘等） |

### 1.4 原则体系结构

本原则体系采用**四维正交结构**，而非简单的层次叠加：

```
         ┌─────────────────────────┐
         │   维度一：系统观         │ ← 控制论 & 系统工程
         │   (世界观与方法论)       │
         ├─────────────────────────┤
         │   维度二：内核观         │ ← 微内核哲学
         │   (内核职责与边界)       │
         ├─────────────────────────┤
         │   维度三：认知观         │ ← 双系统理论
         │   (智能体行为设计)       │
         ├─────────────────────────┤
         │   维度四：工程观         │ ← 乔布斯美学
         │   (代码质量与实现标准)   │
         └─────────────────────────┘
```

四个维度正交但不独立——它们在具体的设计决策中交织，形成 AgentOS 独特的架构 DNA。

## 第 2 章 维度一：系统观

> 维度一回答：AgentOS 作为复杂自适应系统，如何维持动态平衡？

### 2.1 原则 S-1：反馈闭环原则

**内涵**：系统的每一层必须设计完整的感知-决策-执行-反馈闭环。没有反馈的组件是盲目的，没有反馈的系统是僵死的。

**控制论依据**：负反馈消除偏差（稳定性），正反馈放大优势（进化性），前馈预测扰动（预见性）。

**在 AgentOS 中的体现**：

| 反馈类型 | 机制 | 实现位置 |
|---------|------|---------|
| 实时反馈 | 执行结果 → 认知层修正 | `agentos_execution_wait` → `agentos_cognition_process` |
| 轮次内反馈 | 完成节点 → 增量规划器 | execution callback → planner DAG 更新 |
| 跨轮次反馈 | 历史模式 → 策略参数 | L4 模式挖掘 → 认知策略配置热更新 |
| 安全反馈 | 审计日志 → 权限规则 | `domes_audit_record` → `domes_permission_reload` |
| 健康反馈 | 指标采集 → 自动伸缩 | `agentos_sys_telemetry_metrics` → monit_d 告警 |

**实施规则**：
- 任何新增组件必须定义其输入、输出和反馈通道
- 反馈延迟必须明确标注（实时/轮次内/跨轮次）
- 禁止设计"单向"的数据通路——每个输出都必须有对应的消费者

### 2.2 原则 S-2：层次分解原则

**内涵**：复杂系统必须按层次分解，每层只依赖其直接下层的抽象接口，从不越级访问。

**系统工程依据**：层次分解是降低认知复杂度的根本方法。每一层都可以被独立理解、独立验证、独立替换。

**在 AgentOS 中的体现**：

```
syscalls.h → 只依赖 agentos.h（corekern）
cognition.h → 只依赖 agentos.h（corekern）
execution.h → 只依赖 agentos.h（corekern）
memoryrovol.h → 只依赖 agentos.h + layer1~layer4
domes.h → 只依赖 agentos.h（corekern）
```

所有内核头文件的 `#include` 关系严格遵守层次纪律——没有跨层依赖，没有循环引用。

**实施规则**：
- 新模块的头文件只能 `#include` 同层或下层的接口
- 禁止从 `backs/` 直接调用 `atoms/` 内部函数（必须通过 `syscalls.h`）
- 禁止从 `atoms/` 内部模块之间直接耦合（必须通过 `corekern/` 的 IPC）

### 2.3 原则 S-3：总体设计部原则

**内涵**：存在一个只做协调、不做执行的全局决策层。它分解任务但不执行任务，选择策略但不实现策略。

**系统工程依据**：钱学森提出的"总体设计部"概念——复杂系统需要一个超越局部利益的协调者。

**在 AgentOS 中的体现**：认知层（`coreloopthree/cognition`）即是总体设计部：

- 它**不执行**任何具体任务（执行由 `execution/` 完成）
- 它**不存储**任何记忆数据（存储由 `memoryrovol/` 完成）
- 它只做三件事：**理解意图** → **生成计划** → **调度 Agent**

调度官（Dispatcher）进一步体现了这一原则——它只做资源分配与策略决策，通过评分函数选择最优 Agent：

```
Score(agent) = w₁ · (1/cost) + w₂ · success_rate + w₃ · trust_score
```

**实施规则**：
- 决策层与执行层严格分离，通过标准化消息传递
- 策略选择必须基于可量化的评分函数
- 全局优化优先于局部最优

### 2.4 原则 S-4：涌现性管理原则

**内涵**：通过良好的模块交互设计，使系统整体产生超越部分之和的涌现行为——同时抑制负面涌现（如级联故障、资源饥饿）。

**系统工程依据**：涌现性是复杂系统的本质特征。好的架构放大正面涌现（智能、自适应），抑制负面涌现（脆弱、混沌）。

**在 AgentOS 中的体现**：
- **正面涌现**：记忆系统的四层卷载使单个记忆条目经过逐层抽象后涌现出可复用的行为模式
- **正面涌现**：双模型协同使主辅模型的交叉验证涌现出更高的推理可靠性
- **负面涌现抑制**：补偿事务（`compensation.c`）防止任务链中单点失败演变为级联崩溃
- **负面涌现抑制**：权限缓存与热更新防止安全规则变更时的短暂不可用

**实施规则**：
- 每个模块必须文档化其对外涌现的行为
- 必须设计负面涌现的隔离机制（熔断、降级、补偿）
- 禁止依赖未文档化的跨模块涌现行为

## 第 3 章 维度二：内核观

> 维度二回答：内核应该做什么，不应该做什么？

### 3.1 原则 K-1：内核极简原则

**内涵**：内核只保留不可再分的原子机制。一切可以在用户态实现的功能，都必须在用户态实现。

**微内核依据**：Jochen Liedtke 的微内核构造定理——内核的最小职责是 IPC、地址空间管理和线程调度。其他一切皆为服务。

**在 AgentOS 中的体现**：

**`corekern/` 的精确职责边界**：

| 子系统 | 文件 | 职责 | 不负责 |
|--------|------|------|--------|
| IPC | `ipc.h` | 消息通道创建/连接/发送/接收 | 消息内容解析、路由决策 |
| Mem | `mem.h` | 内存分配/释放/池化/泄漏检测 | 数据结构管理、对象生命周期 |
| Task | `task.h` | 线程创建/调度/同步/优先级 | 业务逻辑调度、任务编排 |
| Time | `time.h` | 时钟/定时器/事件 | 超时策略、重试逻辑 |

**实施规则**：
- `corekern/` 新增接口必须通过架构委员会审批
- 新功能优先考虑作为 `backs/` 守护进程 (Daemon) 实现
- 内核接口必须是泛化的——不包含任何业务语义

### 3.2 原则 K-2：接口契约化原则

**内涵**：所有跨模块交互必须通过明确定义的接口进行。接口是模块间的法律契约——签名、语义、所有权、线程安全性、错误处理，缺一不可。

**在 AgentOS 中的体现**：每个公共头文件都包含完整的契约声明：

```c
/**
 * @brief 销毁认知引擎
 * @param engine [in] 引擎句柄（非NULL）
 * @ownership 释放 engine 及其内部资源
 * @threadsafe 否
 * @reentrant 否
 * @see agentos_cognition_create()
 */
AGENTOS_API void agentos_cognition_destroy(agentos_cognition_engine_t* engine);
```

契约包含六个维度：
- **参数方向**：`[in]` 输入、`[out]` 输出、`[in,out]` 双向
- **所有权语义**：谁分配谁释放，引用计数规则
- **线程安全性**：是否线程安全，需要外部同步的范围
- **可重入性**：是否支持重入调用
- **前置/后置条件**：参数约束和返回值保证
- **跨引用**：关联的创建/销毁/查询函数

**实施规则**：
- 所有公共 API 必须包含完整的 Doxygen 契约注释
- 所有权转移必须显式声明
- 线程安全性必须标注（`@threadsafe`）
- ABI 兼容性通过 MAJOR.MINOR.PATCH 版本号管理

### 3.3 原则 K-3：服务隔离原则

**内涵**：所有用户态服务（LLM、市场、监控、调度、工具）作为独立守护进程 (Daemon) 运行，通过系统调用与内核交互。服务之间不直接通信，必须经过内核路由。

**在 AgentOS 中的体现**：

```
backs/llm_d/     → 通过 agentos_sys_* 系统调用与内核通信
backs/market_d/  → 通过 agentos_sys_* 系统调用与内核通信
backs/monit_d/   → 通过 agentos_sys_* 系统调用与内核通信
backs/sched_d/   → 通过 agentos_sys_* 系统调用与内核通信
backs/tool_d/    → 通过 agentos_sys_* 系统调用与内核通信
```

系统调用接口（`syscalls.h`）提供了统一的抽象：

- **任务管理**：`agentos_sys_task_submit/query/wait/cancel`
- **记忆管理**：`agentos_sys_memory_write/search/get/delete`
- **会话管理**：`agentos_sys_session_create/get/close/list`
- **可观测性**：`agentos_sys_telemetry_metrics/traces`

**实施规则**：
- 新增守护进程 (Daemon) 必须通过 `syscalls.h` 与内核交互
- 禁止守护进程 (Daemon) 之间建立直接通信通道
- 服务崩溃必须不影响内核和其他服务的运行

### 3.4 原则 K-4：可插拔策略原则

**内涵**：所有算法选择（规划、协同、调度、遗忘、检索、净化）都设计为可运行时替换的策略接口。策略是系统的"可调参数"——使系统能通过重组适应不同场景。

**在 AgentOS 中的体现**：

| 策略维度 | 接口定义 | 可选实现 |
|---------|---------|---------|
| 规划策略 | `agentos_plan_strategy_t` | 分层规划、反应式规划、反思式规划、ML 规划 |
| 协同策略 | `agentos_coordinator_strategy_t` | 双模型协同、多数投票、加权融合、外部仲裁 |
| 调度策略 | `agentos_dispatching_strategy_t` | 加权调度、轮询调度、优先级调度、ML 调度 |
| 遗忘策略 | `agentos_forget_strategy_t` | 艾宾浩斯曲线、线性衰减、基于访问次数 |
| 执行单元 | `agentos_execution_unit_t` | API、浏览器、代码、数据库、文件、Shell、工具 |

每个策略接口遵循统一的三函数模式：

```c
struct agentos_plan_strategy {
    agentos_plan_func_t plan;       /* 核心算法函数 */
    agentos_plan_destroy_t destroy; /* 资源释放函数 */
    void* data;                     /* 私有数据指针 */
};
```

**实施规则**：
- 新策略必须实现完整的接口三元组（执行、销毁、私有数据）
- 策略切换不能影响正在运行的任务
- 社区贡献的策略必须通过契约验证（`tests/contract/`）

## 第 4 章 维度三：认知观

> 维度三回答：如何让智能体既高效又可靠？

### 4.1 原则 C-1：双系统协同原则

**内涵**：系统的每个认知环节都设计快慢两条路径。System 1 处理高频低复杂度任务，System 2 处理低频高复杂度任务。两者协同而非互斥。

**认知理论依据**：卡尼曼证明人类思维的高效性正来源于双系统的合理分工——System 1 处理 95% 的日常决策，System 2 仅在需要时被激活。

**在 AgentOS 中的体现**：

**认知层的双模型协同**：
- **主思考者（System 2）**：使用最强推理模型进行深度规划与反思
- **辅思考者（System 1）**：使用轻量模型快速响应与交叉验证
- **仲裁机制**：当主辅模型结论不一致时，触发辩论或人工介入

**记忆层的双路径检索**：
- **System 1**：吸引子网络快速模式匹配（O(1) 收敛）
- **System 2**：L4 持久同调深度模式挖掘（O(n²) 但精准）

**实施规则**：
- 新增认知组件必须设计快慢两条路径
- System 1 的输出质量必须有下界保证
- System 2 必须有超时机制，防止无限推理

### 4.2 原则 C-2：增量演化原则

**内涵**：智能体不应一次性规划全部步骤，而应分阶段规划、根据执行反馈动态调整。这避免了长序列任务中的错误累积。

**在 AgentOS 中的体现**：增量规划器（`coreloopthree/cognition/planner/`）：

1. 只生成当前阶段的任务 DAG
2. 根据执行反馈逐步扩展后续阶段
3. 任务失败时可回退到前一阶段，替换失败节点

任务计划以 DAG（有向无环图）表示：

```c
typedef struct agentos_task_node {
    char* task_node_id;                   /* 任务ID */
    char* task_node_agent_role;           /* 需要的Agent角色 */
    char** task_node_depends_on;          /* 依赖的任务ID数组 */
    uint32_t task_node_timeout_ms;        /* 超时时间 */
    uint8_t task_node_priority;           /* 优先级 */
    void* task_node_input;                /* 输入数据 */
    void* task_node_output;               /* 输出数据 */
} agentos_task_node_t;
```

**实施规则**：
- 任务 DAG 的最大深度不应超过 7 层（Miller 定律）
- 每个节点必须有明确的超时和重试策略
- 回退路径必须在规划阶段预定义

### 4.3 原则 C-3：记忆升华原则

**内涵**：记忆不是简单的存储与检索，而是从原始经验中逐层提炼出可复用的知识模式。低层保存事实，高层提炼智慧。

**神经科学依据**：人类海马体-新皮层记忆巩固机制：原始经历 → 特征提取 → 关系绑定 → 图式固化。

**在 AgentOS 中的体现**：MemoryRovol 的四层卷载架构：

| 层级 | 名称 | 输入 | 输出 | 类比 |
|------|------|------|------|------|
| L1 | 原始卷 | 对话/事件/文件 | 带元数据的原始记录 | 海马体 CA3 |
| L2 | 特征层 | L1 原始记录 | 语义向量 + 混合索引 | 内嗅皮层 |
| L3 | 结构层 | L2 特征向量 | 关系图 + 绑定算子 | 海马-新皮层通路 |
| L4 | 模式层 | L3 结构图 | 持久同调 + 可复用规则 | 前额叶皮层 |

**存用分离**：L1 永久保存原始数据（仅追加，永不修改），L2-L4 仅存储索引和特征。检索时通过特征定位，再从 L1 挂载原始数据。

**实施规则**：
- 原始记忆写入后不可修改（immutable）
- 跨层转换必须是单向的（L1 → L2 → L3 → L4）
- L4 模式必须经过验证（`validator.c`）才能注入规则库

### 4.4 原则 C-4：遗忘与固化原则

**内涵**：记忆系统必须同时具备遗忘和固化能力。遗忘释放资源并过滤噪声，固化保留高价值知识并豁免淘汰。

**神经科学依据**：艾宾浩斯遗忘曲线（$R = e^{-\lambda t}$）描述了记忆的自然衰减规律。但高频使用的记忆会被"固化"到长期存储，豁免遗忘。

**在 AgentOS 中的体现**：遗忘引擎（`forgetting.h`）：

- **遗忘策略**：艾宾浩斯指数衰减 / 线性衰减 / 基于访问次数
- **裁剪机制**：低权重记忆被裁剪或归档（`archive.c`）
- **复活机制**：被归档的记忆在相关查询时可被重新激活（`revive.c`）
- **固化机制**：高频使用或高价值记忆被提取为 L4 模式，豁免遗忘

**实施规则**：
- 遗忘阈值必须可配置
- 归档数据必须保留完整备份（不可物理删除）
- 固化操作必须经过验证器确认

## 第 5 章 维度四：工程观

> 维度四回答：代码如何做到既正确又优雅？

### 5.1 原则 E-1：安全内生原则

**内涵**：安全不是附加层，而是内嵌于系统设计的每一个环节。从内核到服务层，从编译时到运行时，安全机制无处不在。

**在 AgentOS 中的体现**：`domes/` 安全穹顶提供四重防护：

| 防护层 | 组件 | 机制 |
|--------|------|------|
| 虚拟工位 | `workbench/` | 进程/容器/WASM 沙箱隔离，资源限额，网络可选 |
| 权限裁决 | `permission/` | YAML 规则引擎，通配符匹配，缓存加速，热更新 |
| 输入净化 | `sanitizer/` | 正则规则过滤，风险等级标注（0-3），可替换或删除 |
| 审计追踪 | `audit/` | 异步写入，日志轮转，结构化查询，全链路记录 |

权限检查的默认立场：**拒绝一切未明确授权的操作**。

**实施规则**：
- 每个新增系统调用必须经过权限检查
- Agent 的工具调用必须经过输入净化
- 所有敏感操作必须记录审计日志
- 安全规则变更必须支持热更新（零停机）

### 5.2 原则 E-2：可观测性完备原则

**内涵**：系统的每一层必须提供结构化的可观测性支持——日志、指标、追踪三位一体，构成完整的系统感知能力。

**在 AgentOS 中的体现**：`utils/observability/` 提供统一的可观测性基础设施：

- **结构化日志**（`logger.h`）：统一格式，TraceID 贯穿，支持日志级别控制
- **指标收集**（`metrics.h`）：计数器、仪表、直方图，支持 Prometheus 导出
- **链路追踪**（`trace.h`）：基于 OpenTelemetry 的请求全链路追踪
- **健康检查**：每个引擎提供 `health_check` 接口，返回 JSON 状态

**实施规则**：
- 每个公共 API 的入口和出口必须记录追踪 span
- 关键路径的延迟必须通过直方图监控
- 错误必须通过错误码（`error.h`）+ 结构化日志双重记录
- 健康检查必须包含依赖项状态（不能只报告自己健康）

### 5.3 原则 E-3：资源确定性原则

**内涵**：每块内存的分配者必须明确其释放者。每个文件句柄的打开者必须明确其关闭者。每个线程的创建者必须明确其销毁者。资源的生命周期必须是确定性的。

**在 AgentOS 中的体现**：

内存管理（`mem.h`）提供完整的生命周期追踪：
- `agentos_mem_alloc_ex` 记录分配位置（文件名 + 行号）
- `agentos_mem_check_leaks` 检测内存泄漏
- 内存池（`pool.c`）提供批量分配/释放

所有权语义通过 Doxygen 契约显式声明：
```c
@ownership out_engine 由调用者负责通过 agentos_cognition_destroy() 释放
@ownership 引擎不接管 fallback 的所有权，调用者仍需负责其生命周期
```

**实施规则**：
- 禁止"分配后不释放"的代码路径
- 所有权转移必须显式文档化
- 每个公共 API 的文档必须说明资源所有权
- 使用 Valgrind 和 AddressSanitizer 进行定期检测

### 5.4 原则 E-4：跨平台一致性原则

**内涵**：内核代码必须在 Linux、macOS、Windows 三个平台上表现一致。平台差异通过抽象层屏蔽，业务代码永远不直接调用平台 API。

**在 AgentOS 中的体现**：

- **线程抽象**（`task.h`）：Windows 使用 `HANDLE`，Unix 使用 `pthread_t`，通过条件编译统一
- **IPC 通道**：跨平台的消息传递抽象
- **构建系统**：CMake 统一管理，21 个 CMakeLists.txt 覆盖所有模块
- **工具链**：支持 GCC、Clang、MSVC 三种编译器

**实施规则**：
- 平台相关代码必须隔离在 `#ifdef` 块中
- 新增平台相关功能必须提供跨平台抽象
- CI/CD 必须在三平台上运行完整测试套件

### 5.5 原则 E-5：接口最小化原则

**内涵**：每个模块只暴露最小的必要接口。接口越少，耦合越低，演进越自由。内部实现细节永远不应泄露到公共头文件中。

**乔布斯美学依据**："简单是最终的复杂。" —— 达·芬奇。乔布斯将这一理念推向极致：iPod 只有一个转盘，iPhone 只有一个 Home 键。

**在 AgentOS 中的体现**：

`agentos.h`（微内核统一入口）只有两个函数：
```c
AGENTOS_API int agentos_core_init(void);
AGENTOS_API void agentos_core_shutdown(void);
```

`memoryrovol.h`（四层记忆系统）只有四个函数：
```c
agentos_memoryrov_init(config, &handle);
agentos_memoryrov_cleanup(handle);
agentos_memoryrov_evolve(handle, force);
agentos_memoryrov_stats(handle, &stats);
```

**实施规则**：
- 公共头文件中的 API 数量应尽可能少
- 内部结构体使用不透明指针（前向声明）
- 模块内部实现细节不应出现在公共头文件中
- 扩展接口通过 `agentos_loop_get_engines` 等回调机制提供

### 5.6 原则 E-6：命名语义化原则

**内涵**：名称即文档。函数名、类型名、变量名必须精确表达其语义。好的命名让代码自解释，减少对注释的依赖。

**乔布斯美学依据**：Apple 的产品命名——"MacBook Air"传达轻薄，"Retina"传达高分辨率。名称本身就是产品故事。

**在 AgentOS 中的体现**：

| 名称 | 语义 |
|------|------|
| `corekern` | 核心内核——不可再分的基础 |
| `coreloopthree` | 三层核心循环——认知-规划-执行的运行时 |
| `memoryrovol` | 记忆卷载——像胶卷一样逐层显影 |
| `domes` | 穹顶——保护性的安全罩 |
| `attractor_network` | 吸引子网络——Hopfield 网络的收敛动力学 |
| `mounter` | 挂载器——将记忆"挂载"到工作空间 |
| `compensation` | 补偿事务——失败时的回滚机制 |

命名规则：
- 类型名：`agentos_<module>_<concept>_t`
- 函数名：`agentos_<module>_<verb>_<object>`
- 宏定义：`AGENTOS_<MODULE>_<CONSTANT>`
- 守护进程 (Daemon)：`<name>_d`

**实施规则**：
- 禁止使用缩写（除非是广泛认可的缩写如 IPC、DAG）
- 禁止使用单字母变量名（循环变量 `i`、`j` 除外）
- 新增命名必须与现有命名风格保持一致

### 5.7 原则 E-7：错误可追溯原则

**内涵**：每个错误都必须携带足够的上下文信息，使调用者能够理解错误原因并采取正确的恢复措施。错误不是终点，而是反馈闭环的起点。

**在 AgentOS 中的体现**：

错误码体系（`error.h`）：
```c
#define AGENTOS_SUCCESS     0    /* 成功 */
#define AGENTOS_EINVAL      -1   /* 无效参数 */
#define AGENTOS_ENOMEM      -2   /* 内存不足 */
#define AGENTOS_EBUSY       -3   /* 资源忙碌 */
#define AGENTOS_ENOENT      -4   /* 不存在 */
#define AGENTOS_EPERM       -5   /* 权限不足 */
#define AGENTOS_ETIMEDOUT   -6   /* 超时 */
#define AGENTOS_ENOTSUP     -9   /* 不支持 */
```

每个错误码对应一个人类可读的描述（`agentos_strerror`），且错误传播路径通过追踪系统记录。

**实施规则**：
- 所有公共 API 必须返回 `agentos_error_t`
- 禁止吞没错误（忽略返回值）
- 错误路径必须释放已分配的资源
- 新增错误码必须经过架构委员会审批

## 第 6 章 原则权衡矩阵

当多个原则发生冲突时，按以下优先级矩阵进行权衡：

| 冲突场景 | 优先原则 | 妥协原则 | 典型处理方法 |
|---------|---------|---------|-------------|
| 安全性 vs 性能 | K-1 内核极简 | E-4 跨平台一致 | 安全校验优先，优化校验算法 |
| 反馈完整性 vs 延迟 | S-1 反馈闭环 | E-5 接口最小化 | 分级反馈：关键路径实时，其他异步 |
| 接口最小化 vs 可观测性 | E-5 接口最小化 | E-2 可观测性完备 | 通过扩展接口提供可观测性 |
| 双系统 vs 资源消耗 | C-1 双系统协同 | E-3 资源确定性 | System 2 按需启动，空闲时释放 |
| 增量规划 vs 全局最优 | C-2 增量演化 | S-3 总体设计部 | 前期规划提供全局约束，增量执行在约束内优化 |
| 记忆完整性 vs 存储效率 | C-3 记忆升华 | E-3 资源确定性 | L1 压缩存储，L2-L4 仅存索引 |
| 策略灵活 vs 接口稳定 | K-4 可插拔策略 | K-2 接口契约化 | 策略接口版本化，旧策略通过适配器兼容 |

### 全局优先级

当上述矩阵未覆盖的冲突出现时，按以下全局优先级裁决：

1. **安全性**（K-1, E-1）—— 系统可以慢，但不能不安全
2. **正确性**（E-3, E-7）—— 系统可以不快，但不能不正确
3. **稳定性**（S-1, K-3）—— 系统可以不新，但不能不稳定
4. **可演进性**（K-4, C-2）—— 系统可以不完美，但不能不可进化
5. **性能**（E-4）—— 在满足以上约束后，追求极致性能

## 第 7 章 案例分析

### 7.1 案例：用户任务的完整生命周期

**场景**：用户输入"帮我分析这份财报并生成摘要"

**原则应用**：

```
[用户输入]
    │
    ▼
[dynamic/ 网关层] ─── E-5 接口最小化（HTTP → Stdio 统一入口）
    │
    ▼
[syscall/ 系统调用] ─── K-3 服务隔离（通过 syscalls.h 标准化）
    │
    ▼
[coreloopthree/ 认知层]
    ├── 意图解析 ─── C-1 双系统协同（快速分类 + 深度理解）
    ├── 增量规划 ─── C-2 增量演化（DAG: 读取文件 → 分析 → 生成摘要）
    ├── Agent 调度 ─── S-3 总体设计部（选择最优 Agent）
    │
    ▼
[domes/ 安全穹顶]
    ├── 权限检查 ─── E-1 安全内生（file:read 权限裁决）
    ├── 输入净化 ─── E-1 安全内生（文件路径净化）
    │
    ▼
[memoryrovol/ 记忆层]
    ├── 上下文挂载 ─── C-3 记忆升华（相关历史记忆挂载到工作空间）
    │
    ▼
[execution/ 执行层]
    ├── 执行任务 ─── E-3 资源确定性（执行单元注册/执行/释放）
    ├── 结果验证 ─── C-1 双系统协同（快速校验 + 深度审核）
    │
    ▼
[反馈闭环]
    ├── 实时反馈 ─── S-1 反馈闭环（执行结果 → 认知层修正）
    ├── 记忆写入 ─── C-4 遗忘与固化（L1 记录，L2 特征提取）
    ├── 审计记录 ─── E-1 安全内生（全链路审计日志）
    │
    ▼
[结果返回]
```

### 7.2 案例：安全规则的运行时热更新

**场景**：管理员更新权限规则，要求 Agent 不再拥有网络访问权限

**原则应用**：
- **K-2 接口契约化**：`domes_permission_reload` 的线程安全性通过契约声明
- **S-1 反馈闭环**：安全反馈——审计日志触发权限规则更新
- **E-1 安全内生**：权限缓存 TTL 机制确保规则变更在毫秒级生效
- **K-4 可插拔策略**：权限引擎本身是可替换的策略组件

### 7.3 案例：新策略的社区贡献

**场景**：社区开发者贡献一个新的 ML 调度策略

**原则应用**：
- **K-4 可插拔策略**：实现 `agentos_dispatching_strategy_t` 三函数接口
- **K-2 接口契约化**：通过 `tests/contract/` 验证接口兼容性
- **E-7 错误可追溯**：策略加载失败时返回明确的 `AGENTOS_ENOTSUP`
- **S-2 层次分解**：新策略文件放在 `coreloopthree/cognition/dispatcher/` 下，不修改其他层

## 第 8 章 架构演进机制

### 8.1 版本管理

本文档与 AgentOS 主版本同步更新。每次架构调整必须更新本文档。

**变更记录要求**：
- 记录变更的架构决策、原因和影响范围
- 评估变更带来的系统性风险
- 更新受影响模块的接口契约

### 8.2 反馈收集

建立架构原则执行的反馈通道：

1. **代码评审反馈**：PR 评审中检查是否符合架构原则
2. **契约测试**：`tests/contract/` 自动验证接口兼容性
3. **架构审计**：定期审查代码结构与原则的符合度
4. **问题回溯**：重大事故回溯是否违背架构原则

### 8.3 持续改进

基于 PDCA 循环持续优化架构原则：

- **Plan**：根据技术演进（如 WASM 沙箱、新的 LLM 架构）提出改进提案
- **Do**：在小范围试点新架构方案
- **Check**：对比试点结果与预期目标
- **Act**：成功经验纳入原则，失败教训形成案例库

### 8.4 例外管理

特殊情况下可能需要偏离架构原则，需遵循以下流程：

1. **例外申请**：提交书面说明，阐述偏离原因和替代方案
2. **影响评估**：架构委员会评估系统性影响
3. **审批决策**：重大例外需架构委员会投票通过
4. **跟踪关闭**：设定例外有效期，到期前必须整改或续期

## 附录 C：设计美学准则

> "设计不仅仅是外观和感觉。设计是它如何工作。" —— Steve Jobs

AgentOS 的代码应当像一件精心设计的产品：

1. **极简入口**：`agentos_core_init` 一个函数启动整个系统
2. **不透明封装**：调用者只看到句柄，不看到实现
3. **契约即法律**：Doxygen 注释不是装饰，是必须遵守的法律
4. **命名即故事**：`memoryrovol`、`coreloopthree`、`domes`——每个名字都讲述一段设计故事
5. **层次即秩序**：`#include` 关系像瀑布一样自上而下，没有逆流
6. **错误即反馈**：每个错误码都是系统在说"这里需要你的注意"
7. **策略即自由**：可插拔的接口让系统像乐高一样可以重组
8. **安全即信任**：穹顶下的每个 Agent 都可以自由工作，因为边界已经被定义

**自检清单**：每次提交代码前，问自己：

- [ ] 这段代码既正确又优雅吗？
- [ ] 这个接口最小化了吗？
- [ ] 这个命名讲了一个好故事吗？
- [ ] 这个资源的生命周期确定吗？
- [ ] 这个错误可以被追溯吗？
- [ ] 这个设计会让八年后的维护者微笑吗？
