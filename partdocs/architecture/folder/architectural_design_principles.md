## 第 1 章 引言

### 1.1 编制目的

本文档为 AgentOS 提供统一的架构设计原则体系。

它不是一份抽象的宣言，而是从项目的每一行代码、每一个头文件、每一次系统调用中提炼出的设计基因。本文档旨在：

1. **统一设计语言**：为所有开发者提供一致的设计决策框架
2. **降低认知负担**：通过四维正交结构，使复杂系统可被理解和推理
3. **保证演进一致性**：确保系统在长期演进中保持架构完整性
4. **提供验证标准**：为代码审查和架构评审提供客观依据

AgentOS 是一个**面向多智能体协作的操作系统内核**：微内核为框架，三层一体认知循环为神经系统，记忆卷载为经验基底，安全穹顶为免疫系统。

它不是构建的又一个 Agent 框架，而是为智能体文明建造的**第一块系统级驱动基石**。

### 1.2 设计起源

#### 1.2.1 问题背景

在 AgentOS 项目启动之前，智能体开发领域面临以下核心挑战：

**架构碎片化问题**：

- 现有 Agent 框架缺乏统一的架构范式
- 各组件之间耦合严重，难以独立演进
- 缺乏系统级的资源管理和安全隔离机制

**认知能力瓶颈**：

- 单一模型难以同时兼顾效率与准确性
- 长序列任务规划存在错误累积问题
- 记忆系统缺乏有效的知识提炼机制

**工程实践缺失**：

- 缺乏可复用的设计模式
- 错误处理和可观测性不足
- 跨平台兼容性差

#### 1.2.2 设计演进

AgentOS 架构设计经历了三个主要阶段的演进：

**阶段一：概念验证（v0.1 - v0.3）**

初步探索微内核架构在智能体系统中的应用可行性。核心发现：

- 微内核的 IPC 机制天然适合多智能体协作
- 进程隔离可以有效防止故障传播

**阶段二：架构成型（v0.4 - v0.9）**

引入三层认知循环和四层记忆系统，形成核心架构骨架。关键突破：

- 双系统认知理论为快慢协同提供了理论基础
- 记忆卷载机制实现了知识的逐层提炼

**阶段三：原则提炼（v1.0 - v2.0）**

从实际代码中反向提炼设计原则，形成四维正交体系。核心成果：

- 16 条核心原则覆盖系统观、内核观、认知观、工程观
- 每条原则都有明确的实施规则和验证方法

#### 1.2.3 设计决策记录

以下关键设计决策塑造了本原则体系：

| 决策编号 | 决策内容 | 理由 | 影响范围 |
|---------|---------|------|---------|
| ADR-001 | 采用微内核架构 | 最小化内核职责，最大化服务隔离 | 全局架构 |
| ADR-002 | 引入双系统协同 | 平衡效率与准确性 | 认知层设计 |
| ADR-003 | 四层记忆卷载 | 实现知识的逐层提炼 | 记忆系统 |
| ADR-004 | 策略可插拔 | 支持运行时算法替换 | 全局扩展性 |
| ADR-005 | 安全穹顶设计 | 安全内生而非附加 | 安全子系统 |

### 1.3 理论根基

#### 1.3.1 工程两论：反馈即生命

> "控制系统依赖反馈来修正偏差。没有反馈的系统是盲目的。"

AgentOS 将反馈设计为贯穿系统每一层的核心机制：

| 反馈层级 | 时间尺度 | 机制描述 | 实现位置 |
|---------|---------|---------|---------|
| 实时反馈 | τ < 100ms | 行动层执行结果返回认知层，修正当前任务 | `agentos_execution_wait` → `agentos_cognition_process` |
| 轮次内反馈 | τ = 任务周期 | 增量规划器根据已完成节点动态扩展 DAG | execution callback → planner DAG 更新 |
| 跨轮次反馈 | τ = 会话周期 | L4 模式层从历史轮次挖掘持久模式 | MemoryRovol L4 → 认知策略配置 |

控制论视角下，AgentOS 是一个**三层嵌套的负反馈系统**：每一层都有独立的感知-决策-执行-反馈闭环，同时层与层之间通过 IPC 通道和系统调用形成跨层反馈通路。

#### 1.3.2 系统工程：层次即秩序

> "复杂系统应按层次分解，每一层只关注自己层次的输入输出。"

AgentOS 采用严格的层次分解：

```
+---------------------------------------------+
|              backs/ 用户态服务               |
|     llm_d · market_d · monit_d · tool_d     |
+---------------------------------------------+
|             dynamic/ 网关层                  |
|          HTTP · WebSocket · Stdio           |
+---------------------------------------------+
|          syscall/ 系统调用接口               |
|    task · memory · session · telemetry      |
+---------------------------------------------+
|            domes/ 安全穹顶                   |
|     workbench · permission · sanitizer      |
+--------------+------------------------------+
| corekern/    |  coreloopthree/              |
| 微内核基础    |  认知-执行三层循环            |
| IPC·Mem·Task |  认知→规划→调度→执行          |
| Time         |                              |
+--------------+------------------------------+
|         memoryrovol/ 四层记忆系统            |
|  L1原始卷 → L2特征层 → L3结构层 → L4模式层    |
+---------------------------------------------+
|             utils/ 公共工具库                |
|  error · logger · metrics · trace · cost    |
+---------------------------------------------+
```

每一层只依赖其下层提供的抽象接口，从不越级访问。这种层次纪律保证了系统各部分可以独立理解、独立测试、独立演进。

#### 1.3.3 双系统认知理论：快慢协同

> "人类思维包含两个系统：快速直觉的 System 1 和缓慢深思的 System 2。"

AgentOS 将双系统理论编码进架构的每一个层次：

| 层次 | System 1（快） | System 2（慢） |
|------|---------------|---------------|
| 认知层 | 辅模型快速分类与交叉验证 | 主模型深度规划与反思调整 |
| 执行层 | 执行单元快速执行 | 补偿事务验证与异常恢复 |
| 记忆层 | 吸引子网络快速模式匹配 | L4 持久同调深度模式挖掘 |
| 安全层 | 净化规则快速过滤 | 权限引擎深度策略裁决 |

双系统不是简单的速度分级，而是**认知带宽的分层管理**：System 1 处理 80% 的常规任务，System 2 集中算力于 20% 的关键决策。

#### 1.3.4 微内核哲学：极简即力量

> "内核只做内核该做的事。其他一切，都是服务。"

AgentOS 遵循严格的微内核架构哲学：

- **`corekern/`** 仅包含四个不可再分的原子机制：IPC（进程间通信）、Mem（内存管理）、Task（任务调度）、Time（时间服务）。总计 7 个头文件、13 个源文件——这是整个系统的最小内核。
- **`coreloopthree/`** 和 **`memoryrovol/`** 作为内核子系统，提供认知循环和记忆演化能力。
- 所有用户态服务（LLM、市场、监控、调度、工具）运行在 `backs/` 中，通过 `syscall/` 的标准化接口与内核交互。
- 安全机制集中在 `domes/` 中，与内核逻辑解耦。

这种分离保证了：内核故障不影响服务可替换性，服务崩溃不会拖垮内核稳定性。

### 1.4 术语定义

| 术语 | 定义 | 英文对照 |
|------|------|---------|
| **原子内核** | 不可再分的微内核基础：IPC、内存、任务、时间 | corekern |
| **三层循环** | 认知→规划→调度→执行的三层认知-行动运行时 | CoreLoopThree |
| **记忆卷载** | 四层递进式记忆系统：原始卷→特征层→结构层→模式层 | MemoryRovol |
| **安全穹顶** | 虚拟工位、权限裁决、输入净化、审计追踪的安全层 | Domes |
| **系统调用** | 用户态服务与内核之间的标准化接口契约 | syscall |
| **守护进程** | 运行在用户态的后台服务，以 `_d` 后缀命名 | Daemon |
| **执行单元** | 可插拔的原子执行器，通过注册表动态加载 | Execution Unit |
| **策略接口** | 可运行时替换的算法抽象（规划/协同/调度/遗忘等） | Strategy |

### 1.5 原则体系结构

本原则体系采用**四维正交结构**，而非简单的层次叠加：

```
         +-------------------------+
         |   维度一：系统观         | <-- 控制论 & 系统工程
         |   (世界观与方法论)       |
         +-------------------------+
         |   维度二：内核观         | <-- 微内核哲学
         |   (内核职责与边界)       |
         +-------------------------+
         |   维度三：认知观         | <-- 双系统理论
         |   (智能体行为设计)       |
         +-------------------------+
         |   维度四：工程观         | <-- 乔布斯美学
         |   (代码的质量与实现标准)  |
         +-------------------------+
```

**四维正交的含义**：

- **正交性**：四个维度相互独立，任一维度的原则不依赖于其他维度
- **交织性**：在具体设计决策中，四个维度同时发挥作用
- **完备性**：四维覆盖了架构设计的所有关键方面

**原则编号规则**：

- **S-X**：系统观原则（System View）
- **K-X**：内核观原则（Kernel View）
- **C-X**：认知观原则（Cognitive View）
- **E-X**：工程观原则（Engineering View）

## 第 2 章 维度一：系统观

> 维度一回答：AgentOS 作为复杂自适应系统，如何维持动态平衡？

### 2.1 原则 S-1：反馈闭环原则

**内涵**：系统的每一层必须设计完整的"感知-决策-执行-反馈"闭环。没有反馈的组件是无效的，没有反馈的系统是有缺陷的。

**控制论依据**：负反馈消除偏差（稳定性），正反馈放大优势（进化性），前馈预测扰动（预见性）。

**在 AgentOS 中的体现**：

| 反馈类型 | 机制 | 实现位置 |
|---------|------|---------|
| 实时反馈 | 执行结果 → 认知层修正 | `agentos_execution_wait` → `agentos_cognition_process` |
| 轮次内反馈 | 完成节点 → 增量规划器 | execution callback → planner DAG 更新 |
| 跨轮次反馈 | 历史模式 → 策略参数 | L4 模式挖掘 → 认知策略配置热更新 |
| 安全反馈 | 审计日志 → 权限规则 | `domes_audit_record` → `domes_permission_reload` |
| 健康反馈 | 指标采集 → 自动伸缩 | `agentos_sys_telemetry_metrics` → monit_d 告警 |

**实施规则**：

1. 任何新增组件必须定义其输入、输出和反馈通道
2. 反馈延迟必须明确标注（实时/轮次内/跨轮次）
3. 禁止设计"单向"的数据通路，每个输出都必须有对应的消费者

**验证方法**：

```c
/**
 * @brief 检查每个公共 API 是否定义了反馈路径
 * @param api_name API 名称
 * @return 是否存在反馈路径
 */
bool has_feedback_path(const char* api_name);
```

### 2.2 原则 S-2：层次分解原则

**内涵**：复杂系统必须按层次分解，每层只依赖其直接下层的抽象接口，从不越级访问。

**系统工程依据**：层次分解是降低认知复杂度的根本方法。每一层都可以被独立理解、独立验证、独立替换。

**在 AgentOS 中的体现**：

```
syscalls.h      --> 只依赖 agentos.h（corekern）
cognition.h     --> 只依赖 agentos.h（corekern）
execution.h     --> 只依赖 agentos.h（corekern）
memoryrovol.h   --> 只依赖 agentos.h + layer1~layer4
domes.h         --> 只依赖 agentos.h（corekern）
```

所有内核头文件的 `#include` 关系严格遵守层次纪律——没有跨层依赖，没有循环引用。

**实施规则**：

1. 新模块的头文件只能 `#include` 同层或下层的接口
2. 禁止从 `backs/` 直接调用 `atoms/` 内部函数（必须通过 `syscalls.h`）
3. 禁止从 `atoms/` 内部模块之间直接耦合（必须通过 `corekern/` 的 IPC）

**验证方法**：

```bash
# 检查头文件依赖层次
python scripts/check_layer_deps.py --strict
```

### 2.3 原则 S-3：总体设计部原则

**内涵**：存在一个只做协调、不做执行的全局决策层。它分解任务但不执行任务，选择策略但不实现策略。

**系统工程依据**："工程两论"提出的"总体设计部"概念：复杂系统需要一个超越局部利益的协调者。

**在 AgentOS 中的体现**：认知层（`coreloopthree/cognition`）即是总体设计部：

- 它**不执行**任何具体任务（执行由 `execution/` 完成）
- 它**不存储**任何记忆数据（存储由 `memoryrovol/` 完成）
- 它只做三件事："**理解意图** → **生成计划** → **调度 Agent**"

调度官（Dispatcher）进一步体现了这一原则，它只做资源分配与策略决策，通过评分函数选择最优 Agent：

```
Score(agent) = w1 * (1/cost) + w2 * success_rate + w3 * trust_score
```

**实施规则**：

1. 决策层与执行层严格分离，通过标准化消息传递
2. 策略选择必须基于可量化的评分函数
3. 全局优化优先于局部最优

### 2.4 原则 S-4：涌现性管理原则

**内涵**：通过良好的模块交互设计，使系统整体产生超越部分之和的涌现行为——同时抑制负面涌现（如级联故障、资源饥饿）。

**系统工程依据**：涌现性是复杂系统的本质特征。好的架构放大正面涌现（智能、自适应），抑制负面涌现（脆弱、混沌）。

**在 AgentOS 中的体现**：

- **正面涌现**：记忆系统的四层卷载使单个记忆条目经过逐层抽象后涌现出可复用的行为模式
- **正面涌现**：双模型协同使主辅模型的交叉验证涌现出更高的推理可靠性
- **负面涌现抑制**：补偿事务（`compensation.c`）防止任务链中单点失败演变为级联崩溃
- **负面涌现抑制**：权限缓存与热更新防止安全规则变更时的短暂不可用

**实施规则**：

1. 每个模块必须文档化其对外涌现的行为
2. 必须设计负面涌现的隔离机制（熔断、降级、补偿）
3. 禁止依赖未文档化的跨模块涌现行为

## 第 3 章 维度二：内核观

> 维度二回答：内核应该做什么，不应该做什么？

### 3.1 原则 K-1：内核极简原则

**内涵**：内核只保留不可再分的原子机制。一切可以在用户态实现的功能，都必须在用户态实现。

**微内核依据**：Jochen Liedtke 的微内核构造定理：内核的最小职责是 IPC、地址空间管理和线程调度，其他一切皆为服务。

**在 AgentOS 中的体现**：

**`corekern/` 的精确职责边界**：

| 子系统 | 文件 | 职责 | 不负责 |
|--------|------|------|--------|
| IPC | `ipc.h` | 消息通道创建/连接/发送/接收 | 消息内容解析、路由决策 |
| Mem | `mem.h` | 内存分配/释放/池化/泄漏检测 | 数据结构管理、对象生命周期 |
| Task | `task.h` | 线程创建/调度/同步/优先级 | 业务逻辑调度、任务编排 |
| Time | `time.h` | 时钟/定时器/事件 | 超时策略、重试逻辑 |

**实施规则**：

1. `corekern/` 新增接口必须通过架构委员会审批
2. 新功能优先考虑作为 `backs/` 守护进程实现
3. 内核接口必须是泛化的——不包含任何业务语义

### 3.2 原则 K-2：接口契约化原则

**内涵**：所有跨模块交互必须通过明确定义的接口进行。接口是模块间的法律契约：签名、语义、所有权、线程安全性、错误处理，缺一不可。

**在 AgentOS 中的体现**：每个公共头文件都包含完整的契约声明：

```c
/**
 * @brief 销毁认知引擎
 * @param engine [in] 引擎句柄（非NULL）
 * @ownership 释放 engine 及其内部资源
 * @threadsafe 否
 * @reentrant 否
 * @see agentos_cognition_create()
 */
AGENTOS_API void agentos_cognition_destroy(agentos_cognition_engine_t* engine);
```

契约包含六个维度：

- **参数方向**：`[in]` 输入、`[out]` 输出、`[in,out]` 双向
- **所有权语义**：谁分配谁释放，引用计数规则
- **线程安全性**：是否线程安全，需要外部同步的范围
- **可重入性**：是否支持重入调用
- **前置/后置条件**：参数约束和返回值保证
- **跨引用**：关联的创建/销毁/查询函数

**实施规则**：

1. 所有公共 API 必须包含完整的 Doxygen 契约注释
2. 所有权转移必须显式声明
3. 线程安全性必须标注（`@threadsafe`）
4. ABI 兼容性通过 MAJOR.MINOR.PATCH 版本号管理

### 3.3 原则 K-3：服务隔离原则

**内涵**：所有用户态服务（LLM、市场、监控、调度、工具）作为独立守护进程运行，通过系统调用与内核交互。服务之间不直接通信，必须经过内核路由。

**在 AgentOS 中的体现**：

```
backs/llm_d/     --> 通过 agentos_sys_* 系统调用与内核通信
backs/market_d/  --> 通过 agentos_sys_* 系统调用与内核通信
backs/monit_d/   --> 通过 agentos_sys_* 系统调用与内核通信
backs/sched_d/   --> 通过 agentos_sys_* 系统调用与内核通信
backs/tool_d/    --> 通过 agentos_sys_* 系统调用与内核通信
```

系统调用接口（`syscalls.h`）提供了统一的抽象：

- **任务管理**：`agentos_sys_task_submit/query/wait/cancel`
- **记忆管理**：`agentos_sys_memory_write/search/get/delete`
- **会话管理**：`agentos_sys_session_create/get/close/list`
- **可观测性**：`agentos_sys_telemetry_metrics/traces`

**实施规则**：

1. 新增守护进程必须通过 `syscalls.h` 与内核交互
2. 禁止守护进程之间建立直接通信通道
3. 服务崩溃必须不影响内核和其他服务的运行

### 3.4 原则 K-4：可插拔策略原则

**内涵**：所有算法选择（规划、协同、调度、遗忘、检索、净化）都设计为可运行时替换的策略接口。策略是系统的"可调参数"——使系统能通过重组适应不同场景。

**在 AgentOS 中的体现**：

| 策略维度 | 接口定义 | 可选实现 |
|---------|---------|---------|
| 规划策略 | `agentos_plan_strategy_t` | 分层规划、反应式规划、反思式规划、ML 规划 |
| 协同策略 | `agentos_coordinator_strategy_t` | 双模型协同、多数投票、加权融合、外部仲裁 |
| 调度策略 | `agentos_dispatching_strategy_t` | 加权调度、轮询调度、优先级调度、ML 调度 |
| 遗忘策略 | `agentos_forget_strategy_t` | 艾宾浩斯曲线、线性衰减、基于访问次数 |
| 执行单元 | `agentos_execution_unit_t` | API、浏览器、代码、数据库、文件、Shell、工具 |

每个策略接口遵循统一的三函数模式：

```c
struct agentos_plan_strategy {
    agentos_plan_func_t plan;       /* 核心算法函数 */
    agentos_plan_destroy_t destroy; /* 资源释放函数 */
    void* data;                     /* 私有数据指针 */
};
```

**实施规则**：

1. 新策略必须实现完整的接口三元组（执行、销毁、私有数据）
2. 策略切换不能影响正在运行的任务
3. 社区贡献的策略必须通过契约验证（`tests/contract/`）

## 第 4 章 维度三：认知观

> 维度三回答：如何让智能体既高效又可靠？

### 4.1 原则 C-1：双系统协同原则

**内涵**：系统的每个认知环节都设计快慢两条路径。System 1 处理高频低复杂度任务，System 2 处理低频高复杂度任务。两者协同而非互斥。

**认知理论依据**：卡尼曼证明人类思维的高效性正来源于双系统的合理分工——System 1 处理 95% 的日常决策，System 2 仅在需要时被激活。

**在 AgentOS 中的体现**：

**认知层的双模型协同**：

- **主思考者（System 2）**：使用最强推理模型进行深度规划与反思
- **辅思考者（System 1）**：使用轻量模型快速响应与交叉验证
- **仲裁机制**：当主辅模型结论不一致时，触发辩论或人工介入

**记忆层的双路径检索**：

- **System 1**：吸引子网络快速模式匹配（O(1) 收敛）
- **System 2**：L4 持久同调深度模式挖掘（O(n^2) 但精准）

**实施规则**：

1. 新增认知组件必须设计快慢两条路径
2. System 1 的输出质量必须有下界保证
3. System 2 必须有超时机制，防止无限推理

### 4.2 原则 C-2：增量演化原则

**内涵**：智能体不应一次性规划全部步骤，而应分阶段规划、根据执行反馈动态调整。这避免了长序列任务中的错误累积。

**在 AgentOS 中的体现**：增量规划器（`coreloopthree/cognition/planner/`）：

1. 只生成当前阶段的任务 DAG
2. 根据执行反馈逐步扩展后续阶段
3. 任务失败时可回退到前一阶段，替换失败节点

任务计划以 DAG（有向无环图）表示：

```c
/**
 * @brief 任务节点结构
 */
typedef struct agentos_task_node {
    char* task_node_id;                   /* 任务ID */
    char* task_node_agent_role;           /* 需要的Agent角色 */
    char** task_node_depends_on;          /* 依赖的任务ID数组 */
    uint32_t task_node_timeout_ms;        /* 超时时间 */
} agentos_task_node_t;
```

**实施规则**：

1. 规划器必须支持增量扩展（不能一次性生成所有节点）
2. 任务失败必须触发回退或重规划
3. 规划深度限制（防止无限递归）

### 4.3 原则 C-3：记忆卷载原则

**内涵**：记忆不是简单的存储与检索，而是从原始经验中逐层提炼出可复用的知识模式。这一过程类比 CNN 的层级特征提取：像素 → 边缘 → 对象 → 场景。

**在 AgentOS 中的体现**：MemoryRovol 四层架构：

| 层级 | 名称 | 输入 | 输出 | 神经科学对应 |
|------|------|------|------|-------------|
| L1 | 原始卷 | 对话/事件/文件 | 带元数据的原始记录 | 海马体 CA3 |
| L2 | 特征层 | L1 原始记录 | 语义向量 + 混合索引 | 内嗅皮层 |
| L3 | 结构层 | L2 特征向量 | 关系图 + 绑定算子 | 海马-新皮层通路 |
| L4 | 模式层 | L3 结构图 | 持久同调 + 可复用规则 | 前额叶皮层 |

**存用分离**：L1 永久保存原始数据（仅追加，永不修改），L2-L4 仅存储索引和特征。检索时通过特征定位，再从 L1 挂载原始数据。

**实施规则**：

1. L1 数据不可变（仅追加，永不修改）
2. L2-L4 索引可重建（丢失后可从 L1 重新生成）
3. 卷载过程必须幂等（多次执行结果一致）

### 4.4 原则 C-4：遗忘机制原则

**内涵**：遗忘不是缺陷，而是记忆系统的核心功能。它裁剪冗余、抑制噪声、保留精华。好的遗忘机制使系统保持高效和可维护。

**在 AgentOS 中的体现**：

**遗忘曲线**：基于艾宾浩斯遗忘曲线，记忆权重随时间衰减：

```
R(t) = e^(-t/tau)
```

其中 tau 是衰减常数（默认 7 天）。

**遗忘策略**：

| 策略 | 公式 | 适用场景 |
|------|------|---------|
| 艾宾浩斯曲线 | R(t) = e^(-t/tau) | 通用场景 |
| 线性衰减 | R(t) = 1 - t/T | 固定生命周期 |
| 基于访问次数 | R(n) = 1 - 1/(n+1) | 热点数据 |

**实施规则**：

1. 遗忘策略必须是可配置的（用户可选择策略）
2. 遗忘阈值必须可调（默认 0.3，低于此值归档）
3. 归档数据必须可复活（通过检索重新激活）

## 第 5 章 维度四：工程观

> 维度四回答：代码如何做到既正确又优雅？

### 5.1 原则 E-1：安全内生原则

**内涵**：安全不是附加层，而是内嵌于系统设计的每一个环节。从内核到服务层，从编译时到运行时，安全机制无处不在。

**在 AgentOS 中的体现**：`domes/` 安全穹顶提供四重防护：

| 防护层 | 组件 | 机制 |
|--------|------|------|
| 虚拟工位 | `workbench/` | 进程/容器/WASM 沙箱隔离，资源限额，网络可选 |
| 权限裁决 | `permission/` | YAML 规则引擎，基于角色的访问控制 (RBAC) |
| 输入净化 | `sanitizer/` | 正则过滤、长度限制、类型检查 |
| 审计追踪 | `audit/` | 全链路追踪，不可篡改的审计日志 |

**实施规则**：

1. 所有外部输入必须经过净化器（`sanitizer/`）
2. 所有权限检查必须通过裁决引擎（`permission/`）
3. 所有执行必须在虚拟工位（`workbench/`）中
4. 所有操作必须记录审计日志（`audit/`）

### 5.2 原则 E-2：可观测性原则

**内涵**：系统必须提供完整的可观测性：指标（Metrics）、追踪（Traces）、日志（Logs）、健康检查（Health）。没有可观测性的系统是黑盒，无法调试、无法优化。

**在 AgentOS 中的体现**：

**四类可观测性数据**：

| 类型 | API | 用途 |
|------|-----|------|
| 指标 | `agentos_sys_telemetry_metrics` | 性能监控、资源使用、业务指标 |
| 追踪 | `agentos_sys_telemetry_traces` | 分布式追踪、调用链分析 |
| 日志 | `logger.h` | 结构化日志、错误诊断 |
| 健康 | `agentos_sys_telemetry_health` | 服务状态、依赖检查 |

**实施规则**：

1. 每个公共 API 的入口和出口必须记录追踪 span
2. 关键路径的延迟必须通过直方图监控
3. 错误必须通过错误码（`error.h`）+ 结构化日志双重记录
4. 健康检查必须包含依赖项状态（不能只报告自己健康）

### 5.3 原则 E-3：资源确定性原则

**内涵**：每块内存的分配者必须明确其释放者。每个文件句柄的打开者必须明确其关闭者。每个线程的创建者必须明确其销毁者。资源的生命周期必须是确定性的。

**在 AgentOS 中的体现**：

**内存管理（`mem.h`）提供完整的生命周期追踪**：

- `agentos_mem_alloc_ex` 记录分配位置（文件:行号）
- `agentos_mem_free` 检查释放位置是否匹配分配位置
- 泄漏检测器在进程退出时报告未释放的内存块
- 内存池（Memory Pool）提供预分配的内存块，避免频繁分配

**实施规则**：

1. 每个资源必须有明确的所有者
2. 资源释放必须通过 RAII 或 cleanup attribute 自动化
3. 禁止隐式资源转移（必须显式声明所有权）

### 5.4 原则 E-4：跨平台一致性原则

**内涵**：内核代码必须在 Linux、macOS、Windows 三个平台上表现一致。平台差异通过抽象层屏蔽，业务代码永远不直接调用平台 API。

**在 AgentOS 中的体现**：

- **线程抽象**（`task.h`）：Windows 使用 `HANDLE`，Unix 使用 `pthread_t`，通过条件编译统一
- **IPC 通道**：跨平台的消息传递抽象
- **构建系统**：CMake 统一管理，21 个 CMakeLists.txt 覆盖所有模块
- **工具链**：支持 GCC、Clang、MSVC 三种编译器

**实施规则**：

1. 平台相关代码必须通过条件编译隔离（`#ifdef _WIN32`）
2. 平台抽象层必须提供统一的 API
3. 跨平台测试必须覆盖所有支持的平台

### 5.5 原则 E-5：命名语义化原则

**内涵**：名称即文档。函数名、类型名、变量名必须精确表达其语义。好的命名让代码自解释，减少对注释的依赖。

**乔布斯美学依据**：Apple 的产品命名——"MacBook Air"传达轻薄，"Retina"传达高分辨率。名称本身就是产品故事。

**在 AgentOS 中的体现**：

| 名称 | 语义 |
|------|------|
| `corekern` | 核心内核——不可再分的基础 |
| `coreloopthree` | 三层核心循环——认知-规划-执行的运行时 |
| `memoryrovol` | 记忆卷载——四层递进式记忆系统 |
| `domes` | 安全穹顶——四重防护的安全层 |
| `backs` | 后台守护进程——用户态服务集合 |

**实施规则**：

1. 函数名采用 `动词_名词` 结构（如 `agentos_task_submit`）
2. 类型名采用 `名词` 结构（如 `agentos_task_t`）
3. 常量名采用 `大写_下划线` 结构（如 `AGENTOS_OK`）
4. 避免缩写（除非是广泛认知的，如 `IPC`、`LLM`）

### 5.6 原则 E-6：错误可追溯原则

**内涵**：每个错误必须可追溯到其根源。错误码、错误消息、调用栈、上下文信息——缺一不可。

**在 AgentOS 中的体现**：

**错误码体系（`error.h`）**：

```c
typedef enum {
    AGENTOS_OK = 0,                  /* 成功 */
    AGENTOS_EINVAL = EINVAL,         /* 无效参数 */
    AGENTOS_ENOMEM = ENOMEM,         /* 内存不足 */
    AGENTOS_EPERM = EPERM,           /* 权限不足 */
    AGENTOS_ETIMEDOUT = ETIMEDOUT,   /* 超时 */
    AGENTOS_EBUSY = EBUSY,           /* 资源忙 */
    AGENTOS_ENOENT = ENOENT,         /* 不存在 */
    AGENTOS_EEXIST = EEXIST,         /* 已存在 */
    AGENTOS_EIO = EIO,               /* I/O 错误 */
    AGENTOS_EFAULT = EFAULT,         /* 地址错误 */
    AGENTOS_EAGAIN = EAGAIN,         /* 重试 */
    AGENTOS_EMSGSIZE = EMSGSIZE,     /* 消息过大 */
    AGENTOS_EPROTO = EPROTO,         /* 协议错误 */
    AGENTOS_ENOSYS = ENOSYS,         /* 功能未实现 */
    AGENTOS_EDEADLK = EDEADLK,       /* 死锁 */
    AGENTOS_EOVERFLOW = EOVERFLOW,   /* 溢出 */
    AGENTOS_ELAST = 255              /* 最后一个错误码 */
} agentos_error_t;
```

**实施规则**：

1. 每个错误必须有唯一的错误码
2. 错误消息必须包含上下文信息（如任务 ID、文件路径）
3. 错误必须通过结构化日志记录
4. 关键错误必须触发告警

### 5.7 原则 E-7：文档即代码原则

**内涵**：文档与代码同步更新。过时的文档比没有文档更糟糕。文档必须作为代码的一部分进行版本控制和审查。

**在 AgentOS 中的体现**：

**文档体系**：

| 文档类型 | 位置 | 内容 |
|---------|------|------|
| 架构文档 | `partdocs/architecture/` | 系统架构、设计原则、核心组件 |
| 规范文档 | `partdocs/specifications/` | 契约规范、编码规范、术语表 |
| API 文档 | `partdocs/api/` | 系统调用、SDK、算法 |
| 指南文档 | `partdocs/guides/` | 快速开始、部署、故障排查 |
| 白皮书 | `partdocs/white_paper/` | 技术白皮书、理论依据 |

**实施规则**：

1. 每个公共 API 必须有 Doxygen 注释
2. 架构变更必须同步更新架构文档
3. 文档必须通过 CI 检查（链接有效性、格式规范）
4. 文档版本必须与代码版本同步

## 第 7 章 原则验证方法

### 7.1 原则符合性检查清单

每个新功能或变更必须通过以下检查清单：

**系统观检查**：

- [ ] 是否设计了完整的反馈闭环？
- [ ] 是否遵守了层次分解原则？
- [ ] 决策层与执行层是否分离？
- [ ] 是否考虑了涌现性管理？

**内核观检查**：

- [ ] 内核职责是否最小化？
- [ ] 接口契约是否完整？
- [ ] 服务是否隔离运行？
- [ ] 策略是否可插拔？

**认知观检查**：

- [ ] 是否设计了双系统协同？
- [ ] 规划是否支持增量演化？
- [ ] 记忆是否分层卷载？
- [ ] 遗忘机制是否合理？

**工程观检查**：

- [ ] 安全是否内生？
- [ ] 可观测性是否完整？
- [ ] 资源生命周期是否确定？
- [ ] 命名是否语义化？
- [ ] 错误是否可追溯？
- [ ] 文档是否同步更新？

### 7.2 原则违反处理流程

当发现原则违反时，按以下流程处理：

1. **记录违规**：在代码审查中标注违反的原则
2. **评估影响**：分析违规对系统的影响程度
3. **制定方案**：提出符合原则的替代方案
4. **实施整改**：修改代码以符合原则
5. **验证合规**：重新检查原则符合性

### 7.3 原则演进机制

原则本身也需要演进。演进流程：

1. **提案**：社区或团队成员提出新原则或修改建议
2. **讨论**：架构委员会组织讨论，评估影响
3. **投票**：架构委员会投票决定是否采纳
4. **文档化**：更新本文档，记录变更原因
5. **公告**：向社区公告原则变更
6. **实施**：在代码库中实施新原则
