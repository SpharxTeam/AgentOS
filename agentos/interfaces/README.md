# 统一接口定义目录

## 📋 目录说明

本目录存放AgentOS所有模块的统一接口定义文件，是双团队（Team A和Team B）协作的标准接口层。所有跨团队协作必须通过本目录定义的接口进行，确保团队间解耦和标准化。

## 🏗️ 目录结构

```
interfaces/
├── platform/              # 平台抽象层（Team A提供，Team B使用）
│   ├── platform.h        # 主接口定义
│   ├── portability.h     # 可移植性辅助
│   └── atomic_compat.h   # 原子操作兼容
├── protocols/            # 协议兼容层（Team B提供，Team A使用）
│   ├── unified_protocol.h
│   ├── adapter_framework.h
│   └── router_interface.h
├── build/               # 构建系统接口（共同维护）
│   ├── cmake_config.h.in
│   ├── dependency_spec.cmake
│   └── platform_detect.cmake
└── sdk/                 # SDK接口（Team B提供）
    ├── c_api.h
    ├── python_stubs.pyi
    └── typescript.d.ts
```

## 👥 团队职责

### Team A（问题修复与跨平台兼容加固）
- **提供平台抽象层接口**：在 `platform/` 目录下定义跨平台兼容接口
- **使用协议兼容层**：使用 `protocols/` 目录定义的协议接口
- **维护构建系统接口**：与Team B共同维护 `build/` 目录下的构建配置

### Team B（框架优化与生态整合）
- **提供协议兼容层接口**：在 `protocols/` 目录下定义协议适配接口
- **提供SDK接口**：在 `sdk/` 目录下定义多语言SDK接口
- **使用平台抽象层**：使用 `platform/` 目录定义的平台接口
- **维护构建系统接口**：与Team A共同维护 `build/` 目录下的构建配置

## 🔧 接口开发流程

### 1. 提案阶段（RFC）
1. 在 `agentos/interfaces/` 目录创建RFC文档（如 `rfc-platform-abstraction.md`）
2. RFC文档必须包含：
   - 接口背景和目的
   - 接口设计详细说明
   - 团队职责划分
   - 版本兼容性策略
   - 测试和验证计划
   - 迁移和升级路径
3. RFC提交双团队评审，获得批准后进入实现阶段

### 2. 接口定义阶段
1. 在对应子目录创建接口头文件（如 `platform/platform.h`）
2. 接口头文件必须包含：
   - 完整的Doxygen风格注释
   - 清晰的API函数签名
   - 详细的使用示例
   - 错误处理约定
   - 线程安全说明
   - 平台兼容性说明
3. 同时提供接口规格文档（如 `platform/SPEC.md`）

### 3. 实现阶段
1. 接口提供团队实现接口功能
2. 接口使用团队编写接口适配代码
3. 双方团队共同编写接口测试用例
4. 所有实现必须通过接口兼容性测试

### 4. 验证阶段
1. 双团队联合评审接口实现
2. 运行完整的接口测试套件
3. 验证跨团队集成场景
4. 生成接口兼容性报告

### 5. 发布阶段
1. 接口正式版本发布
2. 更新接口版本号
3. 更新接口文档
4. 通知所有使用团队

## 📄 接口版本管理

### 版本号规则
- **主版本号（Major）**：不兼容的接口变更
- **次版本号（Minor）**：向下兼容的功能性新增
- **修订号（Patch）**：向下兼容的问题修复

### 版本兼容性要求
1. **主版本变更**：必须提供迁移指南和兼容层
2. **次版本变更**：保证现有接口完全兼容
3. **修订号变更**：不影响接口签名和行为

### 弃用策略
1. 至少提前两个版本标记接口为弃用
2. 提供清晰的替代接口和迁移路径
3. 弃用接口至少保留两个主版本周期

## 🧪 接口测试要求

### 单元测试要求
1. 每个接口函数必须有对应的单元测试
2. 测试覆盖率必须达到85%以上
3. 必须包含边界条件和错误场景测试
4. 必须包含多平台兼容性测试

### 集成测试要求
1. 必须包含跨团队集成测试场景
2. 必须验证接口在实际工作流中的使用
3. 必须包含性能基准测试
4. 必须包含并发安全测试

### 兼容性测试要求
1. 必须验证新接口版本与旧客户端的兼容性
2. 必须验证接口在不同平台上的行为一致性
3. 必须验证接口在异常情况下的健壮性

## 📚 接口文档要求

### 头文件文档
1. 每个接口文件必须有完整的Doxygen注释
2. 每个函数必须有详细的参数说明和返回值说明
3. 必须有清晰的使用示例
4. 必须有线程安全和内存管理说明

### 规格文档
1. 每个接口目录必须有 `SPEC.md` 规格文档
2. 规格文档必须包含接口设计原理和约束条件
3. 必须有明确的性能要求和资源限制
4. 必须有详细的错误处理规范

### 用户指南
1. 每个接口目录必须有 `GUIDE.md` 用户指南
2. 用户指南必须包含快速入门示例
3. 必须有常见问题和解决方案
4. 必须有最佳实践和性能调优指南

## 🔒 接口安全要求

### 输入验证
1. 所有接口必须对输入参数进行严格验证
2. 必须防范缓冲区溢出和注入攻击
3. 必须遵循最小权限原则

### 错误处理
1. 错误信息不得泄露敏感信息
2. 错误处理必须安全可靠
3. 必须提供安全的错误恢复机制

### 安全审计
1. 所有接口代码必须经过安全代码审查
2. 必须定期进行安全漏洞扫描
3. 必须跟踪和修复已知安全漏洞

## 📊 接口质量指标

### 代码质量指标
1. **接口复杂度**：McCabe圈复杂度 ≤ 10
2. **注释密度**：注释行占比 ≥ 30%
3. **重复代码**：重复代码率 ≤ 5%
4. **编译警告**：零编译警告

### 测试质量指标
1. **测试覆盖率**：行覆盖率 ≥ 85%，分支覆盖率 ≥ 80%
2. **测试通过率**：单元测试通过率 100%
3. **性能基准**：性能指标符合规格要求
4. **兼容性测试**：通过所有兼容性测试用例

### 文档质量指标
1. **文档完整性**：所有接口都有完整文档
2. **文档准确性**：文档与实现完全一致
3. **示例可用性**：所有示例代码可运行
4. **更新及时性**：文档随接口变更及时更新

## 🔄 接口变更管理

### 变更申请流程
1. 提交接口变更申请（RFC）
2. 双团队评审变更影响
3. 制定迁移和兼容性方案
4. 获得批准后实施变更

### 变更影响评估
1. 评估对现有用户的影响
2. 评估跨团队集成影响
3. 评估性能和安全性影响
4. 评估测试和文档更新工作量

### 变更实施要求
1. 必须提供向后兼容性支持
2. 必须更新所有相关文档
3. 必须更新所有测试用例
4. 必须进行全面的回归测试

## 🚀 快速开始

### 定义新接口
```bash
# 1. 创建RFC文档
cd agentos/interfaces
echo "# RFC: 新接口设计" > rfc-new-interface.md

# 2. 创建接口头文件
mkdir -p new-interface
cat > new-interface/new_interface.h << 'EOF'
/**
 * @file new_interface.h
 * @brief 新接口定义
 * @version 1.0.0
 */

#ifndef AGENTOS_NEW_INTERFACE_H
#define AGENTOS_NEW_INTERFACE_H

#ifdef __cplusplus
extern "C" {
#endif

/**
 * @brief 示例函数
 * @param param 示例参数
 * @return 示例返回值
 */
int example_function(int param);

#ifdef __cplusplus
}
#endif

#endif // AGENTOS_NEW_INTERFACE_H
EOF

# 3. 创建规格文档
cat > new-interface/SPEC.md << 'EOF'
# 新接口规格

## 设计目标
...

## 接口定义
...

## 使用示例
...
EOF
```

### 使用现有接口
```c
#include "platform/platform.h"
#include "protocols/unified_protocol.h"

// 使用平台抽象接口
platform_init();
platform_create_thread(...);

// 使用协议接口
protocol_connect(...);
protocol_send_message(...);
```
